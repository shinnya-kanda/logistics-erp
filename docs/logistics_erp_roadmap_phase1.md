# 物流ERP 機能一覧・開発優先順位・初期要件定義
作成日: 2026-03-22  
対象: 大友ロジスティクスサービス こまつ営業所向け物流ERP構想  
目的: Cursor / Claude に渡すための初期要件整理

---

## 1. このERPの基本方針

本ERPは、こまつ営業所の現場業務を起点に、以下を実現することを目的とする。

- 現場業務の属人化を減らす
- Excel / CSV / PDF / 手作業を段階的に置き換える
- 出荷・入庫・在庫・請求・配車・歩合計算を将来的に一元化する
- まずは単機能の業務アプリとして成立させ、その後ERPとして統合する
- 将来的な Supabase / PostgreSQL への移行を見据えた設計にする

---

## 2. 開発の前提方針

- 最初からフルERPを作らない
- まずは「現場で本当に使える小機能」を優先する
- マスタとデータ構造を最優先で整える
- UIの見た目再現より、業務ルールの再現を優先する
- コードは AI に実装させ、要件・構造・例外条件は人間が定義する

## 5. なぜこの順番か

### 5-1. マスタ管理を最優先する理由
すべての業務はマスタに依存するため。  
荷主、送付先、集配先、ドライバー、品番、営業所などが曖昧だと、出荷・在庫・請求・配車のすべてが不安定になる。

### 5-2. PDF / CSV取込を早期に実装する理由
現場の元データは PDF / CSV で渡されることが多く、これをデータ化できないと後続の一覧、帳票、在庫、請求に進めないため。

### 5-3. 出庫リスト作成を早期に実装する理由
現場での即効性が高く、成果が見えやすい。  
また、データ設計・マスタ設計・帳票出力設計の検証にもなる。

## 6. Phase 1 の詳細要件

# 6-1. 機能1: マスタ管理

## 目的
出荷・在庫・請求・配車で共通利用する基礎データを一元管理する。

## 対象マスタ
- 荷主マスタ
- 送付先マスタ
- 集配先マスタ
- ドライバーマスタ
- 配送営業所マスタ
- 品番マスタ
- 必要に応じて棚番マスタ、単価マスタへ拡張

## 必須要件
- 一覧表示
- 検索
- 新規追加
- 編集
- 論理削除または無効化
- created_at / updated_at の保持
- CSVインポート / エクスポートを将来的に考慮

## 画面イメージ
- 左メニューにマスタ種別
- 一覧テーブル
- 上部に検索欄
- 右上に新規追加ボタン
- 行ごとに編集ボタン

## 注意点
- コード値の重複を防ぐ
- 表記ゆれを抑える
- 業務で使う名称と内部コードを分ける

---

# 6-2. 機能2: PDF / CSV取込

## 目的
現品票PDFや既存CSVから業務データを読み取り、アプリ内データとして登録する。

## 入力元
- 現品票PDF
- 出荷元CSV
- 客先フォーマットCSV

## 必須要件
- ファイルアップロード
- 取込プレビュー
- 取込失敗行の表示
- 重複チェック
- 取込後の一覧登録
- マスタ突合（品番、送付先、着店等）

## 抽出対象の例
- 発行No
- 品番
- 品名
- 指定納入日
- 数量
- 荷降場
- 送付先
- 着店
- 出庫日
- created / updated

## 重複ルール候補
- 同一発行No + 同一品番 + 同一納入日 は重複登録不可
- ただし運用上必要なら上書き / スキップ / 別登録を選べるようにする

## 注意点
- 元データの列名ゆれに対応できるようにする
- 取込前のカラムマッピング機構を将来的に持てる設計にする
- PDF解析ロジックは画面と分離し、サービス層に置く

---

# 6-3. 機能3: 出庫リスト作成

## 目的
取込済みデータから現場で使用する出庫リストを生成・表示・出力する。

## 必須要件
- 出庫日指定
- 一覧表示
- 納期順表示
- 送付先ごとの集約
- 着店表示
- 明細のPL計算 / 数量集計
- PDF出力
- CSV出力

## 一覧に必要な主な項目
- 出庫日
- 指定納入日
- 発行No
- 品番
- 品名
- 数量
- 荷降場
- 送付先
- 着店
- 棚番
- 備考

## 集約ルール候補
- 同一送付先は1件扱いで集約
- 明細は品番単位で保持
- 出庫リスト表示時に送付先単位・着店単位でまとめる
- delivery_hub をマスタ突合し「出荷リスト着店」に表示する

## 特記事項
- KR / KY など特定ルールの帳票ヘッダ付与がある場合は設定化する
- 作成タブで入力した出庫日は保存データに反映する
- created / updated をデータに保持する

## 8. 状態管理の基本方針

出荷や在庫などの主要データには状態を持たせる。

### 例: shipments.status
- draft
- imported
- confirmed
- completed
- canceled

### 理由
- ERPは「状態遷移」を明示できるかが重要
- 後工程で請求・在庫・配車と連携しやすくなる

---

## 9. Cursor / Claude に出す実装指示の基本方針

以下のルールで実装を進めること。

1. UI、DB、取込処理、帳票出力を責務分離する
2. PDF解析ロジックは services 層に分離する
3. テーブル定義は将来的な PostgreSQL / Supabase 移行を前提にする
4. 全テーブルに created_at / updated_at を付与する
5. 可能なら論理削除または is_active を持たせる
6. コード値の一意制約を適切に設定する
7. バリデーションを必須化する
8. CSV / PDF の取込エラーはユーザーが確認できるUIにする
9. 一覧画面は検索・絞込・並び替えを前提にする
10. 将来的な請求・在庫・配車連携を阻害しない命名にする
